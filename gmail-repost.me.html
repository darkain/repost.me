Also, for the URL shorting we need to allow people to create private or public urls. The difference being that a private url would NOT be tracked in the public statistics and the creator would have the option of requiring a password before the redirect. Only a person with an account would be able to create a private url.

As for account features, I suggest the following:
  • User Controlled Blacklist and Whitelist - This would allow people to click on url's that they may have concerns about w/o the fear of it redirecting them to an area of the internet they don't wish to go. The Whitelist would automatically go forward. The blacklist would display a warning page with the full url displayed. The blacklisted page would never automatically forward but would give the user to manually click the link and move forward.
  • Forwarding Delay - This option would allow the user to setup a delay in the redirect giving them time to view the full address of the link in question. After the delay it would automatically forward. Any items on the whitelist would be exempt from the delay and would be moved forward without delay. Any items on the black list would not be forwarded at all as the blacklisted page would take precedent. 
  • Customized Statistics - This page would allow the user to view various statistics of the URL's they have entered.
I'm sure I'll come up with more ideas later but that's what I have right now. At least for the URL stuff.

so far that very closely matches what i already have.

however, i've put some thought into things and i think i've come up with an idea for the main issue with the system.

i think "private" and "public" URLs should be two separate tables... otherwise, we lose some key optimizations on the URLs.  this way we can keep the key optimizations on the public table.  then not have them for the private table, which would require user interaction anyways, so being slightly slower is okay.

right now, i'm forcing URLs to be unique to help speed things up, otherwise there will be quite a bit of table searching when inserting new items, plus extra PHP code, and table locking.  right now i'm using INSERTs with ON DUPLICATE KEY UPDATE `post`=`post`+1  to quickly get the data into the DB without any table locking at all.  if URLs are no longer unique, we cant do this.

so lets partition out public vs private URLs.  in the public table, URLs are forced unique.  in the private table, they arent (and will require a small amount of locking to ensure no dupes)
